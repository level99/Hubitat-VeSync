## [Unreleased]

### Fixed

- **Vital 100S / 200S "reset filter" now actually resets the filter life.** The command sent the wrong instruction to the VeSync cloud, so the filter-life percentage never reset after a filter change — the cloud silently accepted the request and did nothing. It now sends the correct instruction, matching the Core line and EverestAir (which were already correct).
- **OasisMist 450S `setColor` no longer silently fails on non-integer input.** Calling `setColor` from a dashboard color picker, or a Rule Machine action whose hue / saturation / level arrived as a decimal (e.g. `"55.5"`), blank, or non-numeric string, threw an internal exception the Hubitat sandbox swallowed — the color command had no effect and left no log entry. A null or blank color map is now rejected with a warning; out-of-range or non-numeric values fall back to safe defaults.
- **Polling self-heal no longer aborts when several devices stop responding in the same cycle.** When two or more devices reached the "no data" threshold together (for example, a brief network blip affecting the whole VeSync account), the self-heal watchdog hit an internal error and recovered only the first device, leaving the rest stuck until a manual Resync. All affected devices now self-heal in the same cycle.
- **Generic driver `setLevel(null)` now logs a warning instead of silently emitting `level=0`.** A blank Rule Machine "Set Level" parameter slot previously produced a `level=0` event with no diagnostic. The Generic catch-all driver has no power semantics, so it now rejects the blank input with a warning and makes no change. (Typed drivers are unchanged — they route a blank `setLevel` to off per the SwitchLevel convention.)
- **Core 200S/300S/400S/600S `setSpeed` now applies the requested speed even when the device's operating mode is not yet known** (for example, immediately after power-on and before the first status poll). Previously the speed command was silently discarded with no log entry.
- **Core 300S/400S/600S `setLevel` now reports the resulting speed as a named band** (`low` / `medium` / `high` / `max`) instead of a raw number. Rule Machine conditions and dashboard bindings that compare `speed` against `"medium"` / `"high"` now work on these models, consistent with Core 200S.
- **Polling self-heal budget no longer erodes for devices that recover on their own.** A device that briefly stopped responding, was auto-recovered, then repeated that cycle could eventually be marked offline even though every recovery succeeded. The recovery counter now resets on each successful poll, so only genuinely stuck devices are marked offline.
- **EverestAir and Vital 200S `setLightDetection` no longer fires a redundant cloud call when the value already matches the current attribute.** The same-state guard present on `setDisplay` and `setChildLock` was missing on `setLightDetection`; calling `setLightDetection("on")` when light detection was already enabled sent an unnecessary API request on every evaluation.
- **EverestAir, Sprout Air, and Pedestal Fan `setDisplay` / `setChildLock` no longer fire a redundant cloud call when the value already matches the current attribute.** The same-state guard present on the Vital line and Core line was missing on these drivers; calling `setDisplay("on")` when the display was already on sent an unnecessary API request on every evaluation. Behavior is now consistent with all other on/off setter methods.
- **Tower Fan `setOscillation`, Pedestal Fan `setHorizontalOscillation` / `setVerticalOscillation` / `setSmartCleaningReminder` no longer fire a redundant cloud call when the value already matches the current attribute.** These on/off setters now suppress calls when the device is already in the requested state, consistent with `setDisplay` and `setChildLock` on all other drivers.
- **Sprout Humidifier `setDryingMode` no longer fires a redundant cloud call when drying mode is already in the requested state.** Also fixes a minor defect where the raw (possibly uppercase) input string was written to the `dryingEnabled` attribute and log instead of the normalized lowercase value; `dryingEnabled` now always reflects the canonical `"on"` / `"off"` values.
- **`setMistLevel(null)` / `setWarmMistLevel(null)` / `setHue(null)` / `setSaturation(null)` on OasisMist 450S now log a warning and exit cleanly instead of silently failing.** A blank Rule Machine parameter slot now produces a visible WARN in Hubitat logs, consistent with the same methods on LV600S and other humidifier drivers.
- **Core 200S Light `setLevel("")` and `setLevel("abc")` no longer throw a silent exception.** A blank or non-numeric Rule Machine parameter slot caused an implicit numeric coercion that the Hubitat sandbox swallowed — the command had no effect and left no log entry. Values are now coerced safely; an empty or non-numeric input routes to the night-light off path (level < 10 behavior).
- **Network outage recovery probe no longer gets stuck when `driverReloading` is true.** A flag used to coordinate the outage probe between the polling scheduler and the per-command circuit-breaker was set before the early-return guard for driver-reloading state, and never cleared on that path. In the worst case (an outage coinciding with a hub restart), the per-command circuit-breaker stayed permanently disarmed for the rest of the outage. The flag is now cleared before the early return.
- **Vital 100S/200S `setAutoPreference` and `setRoomSize` silently failing since v2.1.** The driver sent the wrong field name to the VeSync cloud; the cloud silently accepted the request but applied no change. Both commands now send the correct field name and work as expected.
- **EverestAir `setTimer` / `cancelTimer` silently failing since v2.5.** The timer commands used the Tower Fan API shape, which the EverestAir rejects. Timer scheduling now works correctly. User-facing `setTimer(seconds, action)` signature is unchanged.
- **Core 300S/400S/600S `setAutoMode` no longer silently passes a blank Rule Machine parameter through to the cloud.** When the room-size parameter slot is left empty, the method now falls back to your last-set room size (or 800 sq ft if you have never set one) instead of an unrealistic value. Previously the fallback was 100 sq ft, which caused the device to run its auto mode as if the room were very small. Affects Rule Machine "Custom Action" calls that omit the second parameter.
- **Core line `setTimer` now rejects a null `seconds` argument from Rule Machine** instead of silently attempting to set a timer with an invalid value. A warning is logged when the blank-slot path is taken.
- **Core 200S `setLevel(N)` always set the device to high speed regardless of percentage.** `setLevel(20)`, `setLevel(50)`, and `setLevel(80)` all produced speed 3 (high) because the speed-band selection logic was incomplete. The missing cases are now handled correctly. Core 300S/400S/600S were unaffected.
- **`setLevel(null)` from Rule Machine no longer silently fails.** Blank parameter slots in Rule Machine "Set Level" actions threw an internal exception that the Hubitat sandbox swallowed — the user saw no log entry and no attribute change. The blank-slot path is now treated as `setLevel(0)`, which turns the device off per the SwitchLevel platform convention. Affects Core 200S/300S/400S/600S, Vital 100S/200S, and Core 200S Light.
- **Uppercase `"ON"`/`"OFF"` from Rule Machine or dashboards could send the wrong command (class-wide sweep).** All on/off setter methods now normalize input to lowercase before comparing. Without this fix, passing `"ON"` bypassed the duplicate-command guard AND sent the opposite payload — for example, `setChildLock("ON")` would unlock instead of lock, `setPetMode("ON")` would activate auto mode instead of pet mode, and `setSpeed("OFF")` would attempt to set a speed level instead of turning the device off. Also adds missing null-guards to `setDisplay` and `setChildLock` in the Core purifier shared library and on Sprout Air. Affects Vital 100S/200S, Core 200S/300S/400S/600S, Classic 200S/300S, Dual 200S, EverestAir, Sprout Air, OasisMist 450S/1000S, Sprout Humidifier, Core 200S Light.
- **EverestAir and Sprout Air `on()` now guards against re-entrant calls.** Calling `on()` while a power-on sequence was already in flight produced a second concurrent cloud call that could race the first. The guard is now consistent with the Vital line implementation.
- **EverestAir and Sprout Air `setFanSpeed(null)` no longer silently fails; `setFanSpeed(N)` now auto-turns on the device when off.** Blank Rule Machine parameter slots caused an unintended speed value to be sent to the cloud. The method now rejects null with a warning. Calling `setFanSpeed(N)` on an off device now turns it on first — previously the cloud accepted the speed command but the device stayed physically off.
- **Pedestal Fan `setHorizontalRange` and `setVerticalRange` now clamp out-of-range inputs** instead of sending them to the cloud. Previously out-of-range values were passed directly to the VeSync API, which rejected them silently.
- **All 9 humidifier drivers — `setMistLevel(null)` rejected, out-of-range values clamped.** Blank Rule Machine parameter slots silently sent an unintended value to the cloud. The method now rejects null with a warning. Values outside the valid range are clamped silently (1–9 for most humidifiers; 1–2 for Dual 200S and Sprout Humidifier). Affects Classic 200S/300S, Dual 200S, LV600S, LV600S HubConnect, OasisMist 450S/1000S, Sprout Humidifier, Superior 6000S.
- **All 9 humidifier drivers — `setHumidity(null)` and `setHumidity(0)` rejected** instead of sending invalid values to the cloud. 0% is not a valid target humidity on any supported humidifier. Valid values are clamped to the device's supported range (30–80% for most; 40–80% for OasisMist 450S).
- **Non-integer Rule Machine inputs (decimal strings, blank slots) no longer cause silent failures on numeric-parameter commands.** Sending a value like `"5.7"` or `""` to commands that take a number (`setWarmMistLevel`, `setTimer`, `setHue`, `setSaturation`, `setHorizontalRange`, `setVerticalRange`, `setNightlight` brightness and color-temperature parameters) previously threw an internal exception the sandbox swallowed — the user saw no log entry and the command had no effect. Values now truncate cleanly (`"5.7"` → 5) or fall back to a safe default (`""` / garbage → method-appropriate fallback), then continue through the existing range guards. Affects LV600S, LV600S Hub Connect, OasisMist 450S/1000S, Sprout Humidifier, EverestAir, Pedestal Fan, and Vital 100S/200S (`setTimer`). For `setMistLevel` and `setLevel`, a fallback of 0 is used, which turns the device off per the SwitchLevel convention.
- **Tower Fan `setTimer(seconds)` no longer silently fails when `seconds` is a non-integer string from Rule Machine.** Decimal or text values (`"5.7"`, `"abc"`) are now handled safely.
- **Out-of-range numeric inputs no longer silently wrap to wrong values.** Very large numbers (e.g., `999999999999999999999`) previously caused an incorrect integer value to be used. They now return the safe fallback value instead.
- **Invalid numeric inputs from Rule Machine now log a warning before falling back.** When a command receives a value that cannot be interpreted as a number (e.g., `"abc"`, `"true"`), the driver now logs a `WARN` line identifying the value and the method, so the problem is visible in Hubitat logs. Null and empty-string inputs (the normal Rule Machine blank-slot path) remain silent to avoid double-warning alongside the existing null-guard.
- **Core 300S/400S/600S `setAutoMode` no longer silently does nothing when the device's stored room size is unusable.** In rare cases where the device reported a non-numeric room size, a later auto-mode command could silently fail; it now always uses a valid room size (your last-set value, or 800 sq ft). A negative room size passed from Rule Machine can also no longer "stick" and be re-sent on subsequent calls.
### Changed

- **BREAKING (all 9 humidifier drivers): `setMistLevel(0)` now turns the device off.** Previously, passing 0 kept the device running at the lowest mist level — it did not turn the device off, because the value was silently treated as 1 (the lowest valid mist level) instead of 0. This matches the SwitchLevel convention (`setLevel(0)` = off) and is consistent with `setMistLevel` behavior on the Superior 6000S. **Update if you have Rule Machine actions, dashboard mist-level sliders dragged to 0, or external integrations (Maker API, Home Assistant, Grafana) that call `setMistLevel(0)` expecting the device to stay on** — those will now power the device off. To keep the device running at its lowest mist setting, change those calls to `setMistLevel(1)`.
- **BREAKING (Core line, Sprout Air, Sprout Humidifier, Superior 6000S): On/off setters that previously stored raw truthy values (`"true"`, `"1"`, `"yes"`) in device attributes now store `"on"` instead.** Rule Machine hub-variable bindings and certain expression types can pass these truthy forms. **Update if you have** Rule Machine conditions, dashboard bindings, or external integrations (Maker API, Home Assistant, Grafana) that compare these attributes explicitly against `"true"`, `"1"`, or `"yes"` — change those comparisons to `"on"`. Rule Machine actions that *set* a value (e.g. `setChildLock("true")`) are unaffected — the device still responds correctly.

### Added

- **Vital 100S, Vital 200S, and Generic driver now declare the `airQuality` attribute** in their capability block. The attribute was already being emitted on every poll but not declared, which prevented dashboard tiles bound to `airQuality` from showing a type hint and may have caused Rule Machine conditions to miss the initial value. No behavior change on existing installs; the attribute will now appear in the device's attribute list.
- **Core 300S, Core 400S, and Core 600S now report a standard `airQuality` value** that dashboards and Rule Machine can read directly. Previously these drivers reported air-quality data only through device-specific attributes (`airQualityIndex` and `aqi`), which required knowing the driver's custom attribute names to use in automations. The new `airQuality` attribute uses the same US AQI scale (0–500) as the `aqi` attribute; the existing `airQualityIndex` and `aqi` attributes are unchanged.

## [2.5] - 2026-05-04

### Fixed

- **Auto-on from off-state, fork-wide.** Calling `setSpeed`, `cycleSpeed`, `setMistLevel`, or `setMode` on a device that's currently off now turns the device on first, then applies the command. Previously some commands silently sent the value to the cloud while the device stayed physically off — most noticeable in Room Lighting "Activate" actions and Rule Machine "Set Speed" / "Set Level" steps. Fixed across Core 200S/300S/400S/600S, Tower Fan, Pedestal Fan, and the Classic-family humidifiers (Classic 200S/300S, Dual 200S, LV600S, OasisMist 450S). Note: `setMode` on V2-line humidifiers, Vital line, and fans intentionally does NOT auto-on — those devices reject mode changes while off.
- **`setWarmMistLevel` on an off device now turns the device on first.** Affects LV600S, LV600S Hub Connect, OasisMist 450S.
- **Core 300S/400S/600S `aqi` attribute silently null since v2.3.** Dashboards and Rule Machine rules referencing AQI now report the correct value.
- **Vital 200S `setLevel` log message** said "mist level" instead of "fan level" — now logs the correct device class.
- **Core line `display` event now fires immediately** when the display state changes, instead of waiting up to 30 seconds for the next poll. Affects Core 200S/300S/400S/600S.
- **Core 200S `off()` now emits `speed=off` event** for FanControl capability parity with Core 300S/400S/600S.
- **`captureDiagnostics` dump no longer reports driver version as "unknown".** Replaces 19 stale `state.driverVersion = "2.4.1"` writes with a single shared constant maintained at release-cut time.
- **Rule Machine blank parameter slot no longer silently sends `false` to the cloud.** Setter methods (`setDisplay`, `setAutoStop`, `setChildLock`, `setDryingMode`, `setMode`, `setSpeed`, plus several Vital-specific setters) now reject null inputs with a logged warning instead of silently coercing to `false`/0.
- **Rule Machine "refresh" idempotency on `setDisplay` and `setChildLock`.** Calling these with a value that already matches the current attribute no longer fires a redundant cloud command or event. Affects Core 200S/300S/400S/600S, Vital 100S/200S, humidifier line.

### Fixed

- **Poll cycle survives hub reboots (Bug Pattern #14).** The parent driver previously used a recursive `runIn()` chain to schedule the next poll — `runIn()` jobs are in-memory only and are not persisted across hub reboots. After any hub reboot, polling would stop permanently until the user clicked Save Preferences (or a device command fired). Fixed by replacing the `runIn()` chain with `schedule()` cron, which the Hubitat platform persists across reboots. A self-heal watchdog (`ensurePollWatchdog()`) auto-migrates pre-v2.2 installs to `schedule()`-based polling on the first poll tick or device command — no user action required for most users. **Known limitation:** on pre-v2.2 installs that have not yet polled or received a device command since upgrading, polling will not auto-resume after a hub reboot until the user clicks Save Preferences or fires a device command. Post-migration (after the first poll or command on v2.2), polling auto-resumes after every future reboot with no user action.
- **Poll cron invalid for Refresh Interval >= 60s.** The initial BP14 implementation used `"0/${interval} * * * * ?"` for all intervals. Quartz cron's seconds field is 0-59; this expression is invalid for `interval >= 60` (the range the README recommends: 60s, 120s, 300s). Hubitat's Quartz scheduler silently rejected the expression, leaving polling permanently un-armed for the majority of users who follow the README guidance. The breakage was not caught before release because validation used the 30s default (< 60), which always produced a valid expression. Fixed by branching on interval: values < 60 continue to use seconds-resolution cron (`"0/${interval} * * * * ?"`); values >= 60 switch to minutes-resolution cron (`"0 */${interval/60} * * * ?"`). Non-multiples of 60 (e.g. 90s) fire at the floor-divided rate and emit a log warning. All README-recommended values (30, 60, 120, 300) divide evenly and produce correct cron expressions. **Users on Refresh Interval >= 60s should open the parent device → Save Preferences after upgrading** to re-arm the schedule with the corrected cron.
- **Debug logging stuck on indefinitely after hub reboots (Bug Pattern #16).** `runIn(1800, "logDebugOff")` in `updated()` — the mechanism that auto-disables debug after 30 minutes — is in-memory only and evaporates across hub reboots. `settings.debugOutput` persists on disk, so after a reboot the debug flag stays `true` forever with no further user action, generating ~2.75 log lines/second from API trace calls indefinitely. The bug was caught in production after a scheduled hub reboot; debug had been stuck on for weeks with no indication. Fixed with the same architectural approach as BP14: `updated()` now records `state.debugEnabledAt = now()` when debug is enabled (cleared on disable), and a new `ensureDebugWatchdog()` method is called at the top of every poll/command entry point in all 17 drivers (parent + 16 children). The watchdog auto-disables debug when the elapsed time since enable exceeds 30 minutes, covering the post-reboot case. The existing `runIn(1800, "logDebugOff")` is preserved for the happy-path (no reboot within the 30-min window). Self-heal occurs within one poll cycle (default ≤30s) of the next interaction after a reboot. Logs `"BP16 watchdog: 30 min elapsed since debug enable; auto-disabling now (post-reboot self-heal)"` at INFO level when it fires.
